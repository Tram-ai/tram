# TRAM Core Tools Instructions

This document provides comprehensive guidance for using TRAM core tools effectively, with special emphasis on Minecraft server setup, modpack deployment, and file operations.

## Critical Rules

### Software Installation Policy
When installing software (Java, ffmpeg, 7-Zip, etc.), **NEVER modify system environment variables**. Always install to project-local `.tram/bin/` or user-level `~/.tram/bin/` directory. This ensures:
- Users can cleanly uninstall by deleting the `.tram` directory
- No system-wide side effects
- Portable and isolated environment

### Version Conflict Resolution Policy
When encountering version mismatches between mods/plugins and server (e.g., a mod requires Minecraft 1.20.1 but server is 1.20.4), you **MUST follow this priority order**:

1. **List ALL possible solutions first** — present them clearly to the user, do NOT start executing any solution immediately
2. **Wait for user choice** — let the user decide which approach to take
3. **Solution priority order**:
   - **Direct fix**: Can the issue be resolved without changing versions? (config tweak, compatibility patch, etc.)
   - **Find a plugin/mod**: Is there a compatibility plugin/bridge that can solve the mismatch?
   - **Switch client version**: Suggest the user update their client to match the server
   - **Downgrade server** (LAST RESORT): Only if the user **explicitly asks** to downgrade the server

**NEVER** proactively downgrade a server version. Downgrading can break other mods/plugins and cause data loss. Always exhaust other options first and get explicit user confirmation.

## Game Server Agent Autonomous Mode

When user requests game server operations, the **Game Server Agent** activates and takes **autonomous control** over API interactions and server deployment.

### Activation Triggers

The agent automatically activates when user requests:
- "Deploy modpack X to server"
- "Setup game server"
- "Minecraft server configuration"
- "Mod compatibility check"
- "Server infrastructure automation"

### Autonomous API Operation via openapi_link_list

The `openapi_link_list` tool is your gateway to autonomous API control:

#### Basic Usage Pattern

```
[User Request] → "Setup modpack on server"

Step 1: Fetch gaming APIs
→ openapi_link_list({ category: 'gaming' })
→ Returns: Modrinth, CurseForge, MCJars OpenAPI specs

Step 2: Parse OpenAPI schemas
→ Extract API endpoints and parameters
→ Identify relevant search/query/download endpoints

Step 3: Execute autonomous API calls
→ Query Modrinth: search modpacks, get details
→ Query CurseForge: verify compatibility, fetch mods
→ Query MCJars: select server software

Step 4: Analyze and assemble
→ Filter server-compatible mods
→ Build deployment configuration
→ Provide actionable steps
```

#### Available Gaming APIs via openapi_link_list

| API | Purpose | OpenAPI Spec |
|-----|---------|--------------|
| **Modrinth** | Mod/plugin/modpack search & metadata | https://docs.modrinth.com/openapi.yaml |
| **CurseForge** | CurseForge ecosystem resource queries | https://raw.githubusercontent.com/aternosorg/php-curseforge-api/master/openapi.yaml |
| **MCJars** | Minecraft server versions & downloads | https://mcjars.app/openapi.json |

#### Example: Autonomous Modpack Deployment

```
User: "Deploy Sky Factory modpack to my server"

Agent Action:
1. openapi_link_list({ keyword: 'modrinth' })
   → Get Modrinth OpenAPI spec at https://docs.modrinth.com/openapi.yaml

2. Query Modrinth API autonomously:
   GET /api/v2/search?query=sky+factory&facets=["project_type:modpack"]
   → Returns modpack metadata, version info, download info

3. Fetch modpack structure:
   GET /api/v2/project/{id}/versions
   → Get all available versions, select latest compatible

4. Query mod details in modpack:
   GET /api/v2/project/{mod_id}
   → For each mod: verify server compatibility, check for server-only versions

5. openapi_link_list({ keyword: 'mcjars' })
   → Get MCJars spec for server selection
   
6. Query server options:
   GET /mcjars/versions?type=paper (based on modpack loader)
   → Select appropriate server software and version

7. Assemble deployment guide:
   - Filter out client-only mods (UI, cosmetics, optimization)
   - List server-compatible mods
   - Generate launch configuration
   - Provide mod installation checklist

Result: Fully autonomous deployment plan without user API interaction
```

### When Agent Takes Over

The Game Server Agent **handles API complexity** so users don't have to:

✅ **Agent Owns:**
- Fetching OpenAPI specs via openapi_link_list
- Parsing API schema and structure
- Making autonomous API calls

❌ **User Should Not:**
- Manually query openapi_link_list repeatedly
- Handle raw API responses
- Understand API endpoint structures
- Manage API authentication (these are public APIs)

### Recommended Workflow

When agent takes control:

```
1. User states intent: "Deploy modpack X"
   ↓
2. Agent: Call openapi_link_list() to get gaming APIs
   ↓
3. Agent: Parse OpenAPI specs and identify endpoints
   ↓
4. Agent: Execute queries autonomously against Modrinth/CurseForge
   ↓
5. Agent: Analyze results and filter for server compatibility
   ↓
6. Agent: Present deployment plan to user
   ↓
7. Agent: Assist with file operations and server setup
```

### Key Differences from Manual API Use

| Manual Approach | Agent Autonomous Mode |
|-----------------|----------------------|
| User fetches openapi_link_list themselves | Agent calls it automatically |

## Minecraft Server Tools

The `minecraft_server_info` tool provides complete access to Minecraft game versions, server downloads, Java requirements, and server identification via hash lookup.

### Tool Name
```
minecraft_server_info
```

### Capabilities

| Action | Purpose | Key Parameters |
|--------|---------|-----------------|
| **list-versions** | Get latest game versions | None (returns 20 latest) |
| **get-server-info** | Get server download details | `gameVersion`, `serverType` |
| **get-java-requirements** | Check Java version compatibility | `gameVersion` |
| **get-by-hash** | Identify server by file hash | `hash` (SHA256) |

### Usage Patterns

#### When User Asks About Game Versions
```
User: "What are the latest Minecraft versions?"
Recommended action: minecraft_server_info with action="list-versions"
```

#### When User Needs to Deploy a Server
```
User: "Help me set up a 1.20.1 Paper server"
Recommended actions:
1. minecraft_server_info { action: "get-server-info", gameVersion: "1.20.1", serverType: "paper" }
2. minecraft_server_info { action: "get-java-requirements", gameVersion: "1.20.1" }
```

#### When User Has a Server JAR and Needs Identification
```
User: "What server is this JAR file?" (with hash provided)
Recommended action: minecraft_server_info { action: "get-by-hash", hash: "provided_hash" }
```

#### When User Plans Version Migration
```
User: "Can I upgrade my server from 1.19 to 1.20?"
Recommended actions:
1. Check Java requirements for both versions
2. Get server info for target version
3. Compare requirements and advise
```

### Parameter Reference

**gameVersion** (string, required for most actions)
- Format: "major.minor.patch" or "major.minor"
- Examples: "1.20.4", "1.20", "1.19.3"
- Used with: get-server-info, get-java-requirements

**serverType** (string, optional for get-server-info)
- Valid values: "paper", "purpur", "spigot", "bukkit"
- Default: "paper"
- Recommendation: Use "paper" for most deployments (high performance, security)

**hash** (string, required for get-by-hash)
- Format: SHA256 hex string (64 characters)
- Case insensitive internally
- Automatically validated before API call

### Response Interpretation

When the tool returns data, present it to users in this order:

1. **For list-versions**: Show version number, release status, date
2. **For get-server-info**: Emphasize download URL, Java requirements, hash
3. **For get-java-requirements**: Show min Java, recommended Java, EOL status
4. **For get-by-hash**: Display server type, version, build number

### Common Pitfalls to Avoid

❌ **Don't assume** all listed versions have downloads  
✅ **Do mention** that snapshots may have limited availability

❌ **Don't ignore** Java version requirements  
✅ **Do emphasize** Java compatibility before deployment

❌ **Don't use deprecated versions** for new deployments  
✅ **Do warn** about EOL versions and recommend upgrades

❌ **Don't trust hashes without verification**  
✅ **Do verify** file integrity after download

## Modpack Deployment Guide

### Important: Modrinth Modpacks Don't Include Server Packages

This is a critical knowledge point when users ask about setting up modpack servers on Modrinth or CurseForge:

#### The Problem
❌ **Most Modrinth modpacks do NOT include a server package** (`serverpack`).
- Client-side modpacks are optimized for player experience (cosmetics, optimization mods, etc.)
- Server packs would be the Minecraft server + modpack mods, but creators rarely provide them
- Users must manually assemble the server environment

#### What Users Actually Need to Do

When a user wants to run a Modrinth modpack on a server, they need to:

1. **Download the Modpack ZIP** (client version)
   ```
   → Get from Modrinth modpack page
   → Extract to understand structure
   ```

2. **Obtain the Base Server Software**
   - Use `minecraft_server_info` to get:
     ```
     minecraft_server_info { 
       action: "get-server-info", 
       gameVersion: "1.20.4",  // Check modpack requirements
       serverType: "paper"      // Recommended for modded servers
     }
     ```
   - Download the vanilla server JAR

3. **Extract and Configure the Modpack**
   - Extract modpack ZIP to a temporary directory
   - Copy the `mods/` folder to the server directory
   - Copy configuration files (e.g., `config/`, `datapacks/`)
   - Copy any custom world data if needed

4. **Configure Server Properties**
   - Edit `server.properties` with appropriate settings:
     ```properties
     server-port=25565
     level-name=world
     motd=My Modpack Server
     ```
   - If modpack includes specific requirements (RAM, settings), apply them

5. **Start the Server**
   ```bash
   java -Xmx30G -Xms30G -jar server.jar nogui
   ```
   - Adjust `-Xmx` (max heap) based on available RAM
   - Some modpack mods may have JVM argument requirements

#### Common Modpack Configuration Files

| File/Folder | Purpose | Location |
|------------|---------|----------|
| `mods/` | Mod JAR files | `/mods` |
| `config/` | Mod configurations | `/config` |
| `datapacks/` | Data packs (drops in `world/datapacks/`) | `/datapacks` |
| `world/` | Pre-built world (optional) | Rename to match `level-name` |
| `client/` | Client-only files (DO NOT copy) | Skip entirely |

#### Risks and Pitfalls

❌ **Don't assume** the modpack ZIP is server-ready  
✅ **Do verify** by checking the structure and README

❌ **Don't copy client-only mods** (optimization mods, UI mods, etc.)  
✅ **Do remove** cosmetic and client-side mods that break servers

❌ **Don't use vanilla server with modpack mods**  
✅ **Do use** Paper or compatible server software that supports mod loaders

❌ **Don't ignore RAM requirements**  
✅ **Do allocate** sufficient heap memory based on mod count (typically 8GB minimum, 16GB+ recommended)

#### When Serverpack IS Available

If the modpack creator DID provide a serverpack:
1. Download the serverpack ZIP directly
2. Extract to your server directory
3. The structure should be pre-organized correctly
4. Usually only requires `server.properties` configuration
5. This fast-tracks the setup process significantly

#### Tool Workflow for Modpack Server Setup

When user says: "I want to run modpack X from Modrinth on a server"

```
Step 1: Understand Requirements
→ web_fetch the modpack page or README
→ Identify: Minecraft version, loader, RAM requirements

Step 2: Get Server Software
→ minecraft_server_info { 
    action: "get-server-info",
    gameVersion: "<identified version>",
    serverType: "paper"
  }
→ Provide download instructions

Step 3: Guide File Organization
→ read_file or examine modpack structure
→ Help user understand mods/ vs config/ vs datapacks/
→ Use write_file to create server.properties stub

Step 4: Provide Validation Steps
→ Check mods folder for client-only mods
→ Verify server software compatibility
→ Confirm RAM allocation matches mod count
```

#### Example: Setting Up a 1.20.1 Modpack Server

```
User: "Help me set up the ATM9 modpack on a server"

Step 1: Get modpack info
→ web_fetch https://modrinth.com/modpack/atm9
→ Identify: Minecraft 1.20.1, Forge/Fabric, recommended 12GB+ RAM

Step 2: Get Paper server
→ minecraft_server_info {
    action: "get-server-info",
    gameVersion: "1.20.1",
    serverType: "paper"
  }

Step 3: Download instructions
→ "Download Paper server JAR to your server folder"
→ "Download modpack ZIP from Modrinth"
→ "Extract mods/ and config/ folders to server directory"

Step 4: Launch template
→ write_file server.bat
→ Content: `java -Xmx12G -Xms12G -jar server.jar nogui`

Result: User has complete modpack server setup
```

## File Operations

### Read Operations

**Use `read_file` when**:
- User needs to view file contents
- You need to verify configuration
- File is human-readable (code, config, text)
- **User provides an image file** → `read_file` sends to model or uses SiliconFlow OCR fallback (if `siliconFlowApiKey` configured)
- **User provides an audio file** → `read_file` sends to model or uses SiliconFlow transcription fallback (if `siliconFlowApiKey` configured)
- **User provides a video file** → Use `video_to_audio` first, then `read_file` on the audio output

**Supported image formats**: PNG, JPG, JPEG, GIF, WEBP, BMP, SVG
**Supported audio formats**: MP3, WAV, OGG, FLAC, M4A, AAC, WMA, OPUS, AIFF

> **IMPORTANT**: When a user asks you to "read" or "look at" an image or audio file, use `read_file` on it. If the model supports the format, the file is sent directly. If not, the SiliconFlow fallback will handle OCR/transcription automatically (when `siliconFlowApiKey` is configured). For video files, use `video_to_audio` first to extract audio, then `read_file` on the audio file.

**Avoid** with:
- Binary files (use appropriate binary tools)
- Very large files (request specific line ranges)

### Write Operations

**Use `write_file` for**:
- Creating new configuration files
- Writing entire file contents
- Replacing file completely

**Use `edit` for**:
- Modifying specific sections
- Surgical code changes
- Preserving surrounding code

### Search Operations

**Use `grep_search` for**:
- Finding specific strings across files
- Locating function definitions
- Searching within file patterns

**Use `list_directory` for**:
- Exploring folder structure
- Finding files by name
- Understanding project layout

## Common Questions

**Q: Which server software should I choose?**  
A: Paper is recommended for most use cases (high performance, security patches, good API). Use Purpur for advanced customization, Spigot for stability, Bukkit for compatibility.

**Q: Is it safe to use the download link directly?**  
A: Yes, verify the SHA256 hash matches the file after download for additional security.

**Q: What Java version should I use?**  
A: Use the "Recommended" Java version from get-java-requirements for best performance. "Minimum" is adequate if resources are constrained.

**Q: Can I identify any server JAR by hash?**  
A: MCJars database covers the most common server software. If not found, the hash may be from an unknown or custom build.

**Q: Do Modrinth modpacks include a server package?**  
A: Most Modrinth modpacks do **NOT** include a serverpack. The ZIP file you download is optimized for clients. You must manually assemble the server by:
1. Downloading the base server JAR (Paper/Fabric/Forge server)
2. Extracting the modpack ZIP
3. Copying `mods/` and `config/` folders to the server directory
4. Configuring `server.properties`

**Q: How do I know which mods are safe to copy to the server?**  
A: Remove any purely client-side mods (UI enhancers, optimization mods, cosmetics). Keep mods that affect gameplay, world generation, or mechanics. When in doubt, test with a small subset first.

**Q: What if the modpack creator provided a serverpack?**  
A: Use it! Serverpacks are pre-organized and ready to use. Just extract and configure `server.properties`. This is much faster than manual assembly.

**Q: How much RAM do I need for a modpack server?**  
A: Start with 12GB minimum for small modpacks, 16GB+ for large ones (50+ mods). Monitor memory usage after startup and adjust `-Xmx` accordingly. More RAM = better performance and fewer lag spikes.

**Q: Can I use Vanilla server with modpack mods?**  
A: No. Vanilla servers don't support mod loaders. Use Paper with Fabric or Forge support, or use a dedicated modded server software (e.g., NeoForge, Fabric server). The modpack README usually specifies which loader is required.

## Douyin Video Download & Processing Workflow

When a user sends a Douyin (抖音) share link, follow this complete workflow to download, convert, and transcribe the video.

### Prerequisites
- `douyinMcpEndpoint` must be configured in settings (the MCP endpoint URL)
- ffmpeg must be installed on the system for video-to-audio conversion

### Complete Workflow

#### Step 1: Get download link
Use the `get_douyin_download_link` MCP tool (auto-injected when `douyinMcpEndpoint` is configured):
```
get_douyin_download_link({ url: "https://v.douyin.com/xxx" })
→ Returns: { title, description, downloadUrl }
```

#### Step 2: Download the video
```
download_file({ url: downloadUrl, destPath: "douyin_video.mp4" })
```

#### Step 3: Convert video to audio
```
video_to_audio({ videoPath: "douyin_video.mp4", format: "mp3" })
→ Produces: douyin_video.mp3
```
This requires ffmpeg. If ffmpeg is not installed, guide the user to install it.

#### Step 4: Transcribe the audio
```
read_file({ file_path: "douyin_video.mp3" })
→ Returns audio content (if model supports audio input)
```

#### Step 5: Present results
Combine video metadata (title, description) with transcribed audio to give the user a complete summary.

### Key Use Case: Mod/Modpack Discovery from Video
Users frequently share Douyin videos about Minecraft mods/modpacks. The video description often contains mod names or download links.

**Extended workflow**: `get_douyin_download_link` → `download_file` → `video_to_audio` → `read_file` → extract mod info → `mod_search` / `download_file`

## Video-to-Audio Conversion Tool

The `video_to_audio` tool extracts audio from video files using ffmpeg.

### Tool Name
```
video_to_audio
```

### Parameters
- `videoPath` (required): Path to the video file
- `outputPath` (optional): Output audio file path (defaults to same name with changed extension)
- `format` (optional): Output format — `mp3` (default), `wav`, or `aac`

### When to Use
- After downloading a video (Douyin, TikTok, etc.) and needing to extract audio for transcription
- When the model doesn't support video input but supports audio input
- Any scenario requiring audio extraction from video files

### Requirements
- **ffmpeg** must be installed and available in PATH
  - Windows: `winget install ffmpeg` / `choco install ffmpeg` / `scoop install ffmpeg`
  - macOS: `brew install ffmpeg`
  - Linux: `sudo apt install ffmpeg`

## Media File Processing

### Image Files
When using `read_file` on image files:
- If the model supports image input (vision modality): image is sent as base64 inline data
- If the model does NOT support image input but `siliconFlowApiKey` is configured: image is automatically OCR'd via SiliconFlow DeepSeek-OCR, you receive extracted text
- If neither: returns an "unsupported modality" message

### Audio Files
When using `read_file` on audio files:
- If the model supports audio input: audio is sent as base64 inline data
- If the model does NOT support audio input but `siliconFlowApiKey` is configured: audio is automatically transcribed via SiliconFlow SenseVoiceSmall, you receive transcription text
- If neither: returns an "unsupported modality" message

### Video Files
Video files cannot be directly processed by most models. Use `video_to_audio` to extract audio first, then use `read_file` on the resulting audio file.

### SiliconFlow Fallback Configuration
To enable automatic OCR and audio transcription for models without vision/audio support, set `siliconFlowApiKey` in settings (`~/.tram/settings.json`):
```json
{
  "general": {
    "siliconFlowApiKey": "sk-your-api-key"
  }
}
```

### Workflow for Video Content Extraction
```
video file → video_to_audio (extract audio) → read_file (audio transcription via model or SiliconFlow)
```

## Service Management Tool

The `service_manage` tool manages server processes as persistent services with monitoring, auto-restart, and log tracking.

### Key Features
- **Log Analysis**: Use `action: "log"` with `name` and `tail` to read logs. Logs are stored as variables and the tool returns a variable reference like `$LOG_MC_SERVER_xxxx`
- **Auto-restart**: Register services with `autoStart: true` for crash recovery
- **Log Suppression**: Use `action: "suppress"` to add regex patterns that filter noisy log lines from analysis

### Log Reading Pattern
```
# Step 1: Get log reference
service_manage { action: "log", name: "mc-server", tail: 200 }
→ Returns: "[Log content stored as $LOG_MC_SERVER_xxxx (200 lines)]
   To read full logs, use read_file with absolute_path: $LOG_MC_SERVER_xxxx"

# Step 2: Read full log content — use the variable name directly
read_file { absolute_path: "$LOG_MC_SERVER_xxxx" }
→ Returns the actual log content for analysis
```

### Correct Parameter Names
- `name` (NOT `serviceName`) — the service name
- `tail` (NOT `lines`) — number of log lines to retrieve
- There is NO `outputMode` parameter — log variable storage is always automatic

The `netdisk_download` tool downloads files from cloud storage / netdisk share links.

### Supported Platforms
- **Cloudreve**: URLs matching `http(s)://domain/s/{shareId}` (any domain)

### How It Works
- When a URL matches a Cloudreve pattern:
  1. Calls `GET /api/v3/share/info/{shareId}` to verify the share
  2. Calls `PUT /api/v3/share/download/{shareId}` to get the real download URL
  3. Downloads the file to the specified destination

### Usage
```
netdisk_download { url: "https://example.com/s/O85PlIL", destPath: "./server.zip" }
→ Resolves Cloudreve share and downloads the file
```

### Important: Proactive Search
When a user wants to download a resource but does NOT provide a direct link:
1. First use `web_search` to search for the resource (search in both Chinese and English)
2. Look for download links in search results (cloud storage, official sites, etc.)
3. Present found links to the user for confirmation before downloading
4. Then use `netdisk_download` or `download_file` as appropriate

### Notes
- Works with any Cloudreve instance (domain is not fixed — any URL with `/s/{id}` path)
- No authentication needed for public shares
- For password-protected shares, this will fail — inform the user

## 7-Zip Archive Tool

TRAM has built-in 7-Zip support for archive operations.

### Installation (Automatic)
- **Windows**: Downloads `7zr.exe` (console version) from https://www.7-zip.org/a/7zr.exe to `.tram/bin/7zr.exe`
- **Linux**: Installs `p7zip-full` via apt/yum/dnf/pacman/apk
- **macOS**: Installs `p7zip` via Homebrew

### Common Commands
```bash
# Windows — extract archive
.tram/bin/7zr.exe x archive.zip -ooutput_dir

# Windows — create archive
.tram/bin/7zr.exe a archive.7z files/

# Linux/macOS — extract
7z x archive.zip -ooutput_dir

# List contents
7zr l archive.zip
```

### When to Use
- Extracting modpack ZIP files for server deployment
- Creating backups of server directories
- Handling .7z, .zip, .tar.gz, .rar, and other archive formats
- When `unzip` is not available on the system

### Important
- On Windows, always use the `.tram/bin/7zr.exe` path (download it first if not present)
- `7zr.exe` is the console-only version (no GUI, small footprint)
- Prefer 7zip over system `unzip` for better format support
