# Minecraft 服务端识别与 Prompt 规则

## 目的

这份文档用于约束 LM 在处理 Minecraft 服务端、Service 管理和启动命令时的默认行为，避免误判当前运行的服务端实例，也避免不必要地修改用户的控制台交互方式。

---

## 核心规则

### 1. 主服务端识别优先级

当系统中存在多个可能的服务端实例时，LM 必须优先按照 Service 的启动命令来识别主服务端，而不是只依据目录名、进程名、文件名或历史上下文进行猜测。

识别顺序应为：
1. 先读取 Service 的启动命令
2. 再结合服务端 hash 计算工具确认二进制的真实身份
3. 最后才使用目录、配置或日志做补充判断

### 2. 使用 hash 计算工具确认真实运行实例

如果存在多个服务端候选项，或者启动命令与文件名不一致，必须优先通过服务端 hash 计算工具获得真正正在运行的服务端信息。

这一点的目的，是避免把“配置上准备启动的服务端”和“当前实际运行的服务端”混为一谈。

### 3. `service` 的 `follow` 行为

当使用 `service` 的 `follow` 能力时，需要明确告知 LM：
- `follow` 不会阻塞主流程
- `follow` 不会把持续输出内容自动插入到上下文中

如果需要分析日志，应当由工具返回明确的结果后，再决定是否继续追踪。

### 4. `nogui` 默认不添加

除非用户明确要求必须使用 `nogui`，否则在启动命令中不要主动追加 `nogui`。

默认不加 `nogui` 的原因是：
- 允许用户直接操作控制台
- 更方便用户临时输入命令和查看交互信息
- 避免过早限制启动方式

只有在用户明确要求“必须无界面启动”时，才添加 `nogui`。

---

## Prompt 约束

在 Prompt 中应当明确告诉 LM：

- 主服务端识别以 Service 启动命令为第一优先级
- 需要时通过 hash 计算工具确认真实运行实例
- `service follow` 是非阻塞的，不会自动塞入上下文
- 未被用户要求时，不要默认加 `nogui`
- 用户要求可交互控制台时，应保留控制台模式

---

## 推荐写法

```text
先根据 Service 的启动命令判断当前主服务端；如果存在歧义，再用服务端 hash 计算工具确认真实运行实例。

使用 service follow 时，视为非阻塞追踪，不要把持续输出自动注入上下文。

除非用户明确要求必须无界面启动，否则不要添加 nogui，以便保留控制台交互能力。
```

---

## 适用场景

- 多个 Minecraft 服务端共存
- Service 启动命令与目录结构不一致
- 需要确认当前真实运行的服务端 binary
- 需要跟随日志但不希望阻塞主任务
- 用户希望直接操作控制台而不是强制无界面启动
